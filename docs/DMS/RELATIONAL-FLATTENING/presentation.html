Document Flattening to Relational Tables

Executive Summary

  • Objective: Flatten JSON documents stored in DMS Documents table into relational tables for backwards compatibility with legacy ODS/API
  • Approach: Dynamic extension registration system with schema separation
  • Key Innovation: Zero-modification core logic for new extensions
  • Benefits: Maintains document/relational duality while supporting unlimited extensions
Decision Required: Approve implementation of comprehensive flattening design with dynamic extension framework

Dynamic Extensions

Extension Registration System

Registry Tables

-- Extension configuration registry CREATE TABLE dms.extension_registry ( extension_name VARCHAR(100) PRIMARY KEY, schema_name VARCHAR(63) NOT NULL, is_enabled BOOLEAN NOT NULL DEFAULT true, priority INT NOT NULL DEFAULT 100, created_at TIMESTAMP NOT NULL DEFAULT NOW() ); -- Resource flattening function registry CREATE TABLE dms.resource_flattening_registry ( id SERIAL PRIMARY KEY, resource_name VARCHAR(256) NOT NULL, extension_name VARCHAR(100), -- NULL for core Ed-Fi flattening_function VARCHAR(500) NOT NULL, target_table VARCHAR(500) NOT NULL, priority INT NOT NULL DEFAULT 100 ); -- Extension flattening for core resources CREATE TABLE dms.extension_flattening_registry ( id SERIAL PRIMARY KEY, core_resource_name VARCHAR(256) NOT NULL, extension_name VARCHAR(100) NOT NULL, flattening_function VARCHAR(500) NOT NULL, priority INT NOT NULL DEFAULT 100 );

Extension Design Principles

  • Loose Coupling: Core flattening logic doesn't know about specific extensions
  • Dynamic Configuration: Extensions configured at deployment time
  • No Core Modifications: Adding extensions requires no changes to core logic
  • MetaEd Generated: All extension artifacts generated by MetaEd

Technical Implementation Details

Dynamic Document Flattening

CREATE OR REPLACE FUNCTION dms.flatten_document( p_document_id BIGINT, p_document_partition_key SMALLINT ) RETURNS VOID AS $$ DECLARE v_document RECORD; v_flattening_function VARCHAR(500); BEGIN -- Get document SELECT * INTO v_document FROM dms.Document WHERE Id = p_document_id AND DocumentPartitionKey = p_document_partition_key; -- Look up flattening function from registry SELECT rf.flattening_function INTO v_flattening_function FROM dms.resource_flattening_registry rf LEFT JOIN dms.extension_registry er ON rf.extension_name = er.extension_name WHERE rf.resource_name = v_document.ResourceName AND (rf.extension_name IS NULL OR er.is_enabled = true) ORDER BY rf.priority DESC LIMIT 1; -- Dynamically execute the registered function IF v_flattening_function IS NOT NULL THEN EXECUTE format('SELECT %s($1, $2)', v_flattening_function) USING v_document, v_document.EdfiDoc; END IF; END; $$ LANGUAGE plpgsql;

Core Entity Table Example

CREATE TABLE edfi.Student ( -- DMS Reference DocumentId BIGINT NOT NULL, DocumentPartitionKey SMALLINT NOT NULL, DocumentUuid UUID NOT NULL, -- Natural Key StudentUniqueId VARCHAR(32) NOT NULL, -- Core Attributes FirstName VARCHAR(75) NOT NULL, LastSurname VARCHAR(75) NOT NULL, BirthDate DATE NOT NULL, -- ... other Ed-Fi fields CONSTRAINT Student_PK PRIMARY KEY (DocumentPartitionKey, DocumentId), CONSTRAINT Student_UK_StudentUniqueId UNIQUE (StudentUniqueId) ); CREATE TABLE edfi.StudentAddress ( StudentDocumentId BIGINT NOT NULL, StudentDocumentPartitionKey SMALLINT NOT NULL, AddressTypeDescriptorId INT NOT NULL, StreetNumberName VARCHAR(150) NOT NULL, City VARCHAR(30) NOT NULL, StateAbbreviationDescriptorId INT NOT NULL, PostalCode VARCHAR(17) NOT NULL, CONSTRAINT StudentAddress_PK PRIMARY KEY ( StudentDocumentPartitionKey, StudentDocumentId, AddressTypeDescriptorId ), CONSTRAINT StudentAddress_FK_Student FOREIGN KEY ( StudentDocumentPartitionKey, StudentDocumentId ) REFERENCES edfi.Student(DocumentPartitionKey, DocumentId) );

Student Flattening Function Example

CREATE OR REPLACE FUNCTION dms.flatten_edfi_student( p_document dms.Document, p_edfi_doc JSONB ) RETURNS VOID AS $$ DECLARE v_address JSONB; v_address_type_descriptor_id INT; v_state_descriptor_id INT; BEGIN -- Insert/Update core Student record INSERT INTO edfi.Student ( DocumentId, DocumentPartitionKey, DocumentUuid, StudentUniqueId, FirstName, LastSurname, BirthDate ) VALUES ( p_document.Id, p_document.DocumentPartitionKey, p_document.DocumentUuid, p_edfi_doc->>'studentUniqueId', p_edfi_doc->>'firstName', p_edfi_doc->>'lastSurname', (p_edfi_doc->>'birthDate')::DATE ) ON CONFLICT (DocumentPartitionKey, DocumentId) DO UPDATE SET StudentUniqueId = EXCLUDED.StudentUniqueId, FirstName = EXCLUDED.FirstName, LastSurname = EXCLUDED.LastSurname, BirthDate = EXCLUDED.BirthDate; -- Clear existing addresses for this student DELETE FROM edfi.StudentAddress WHERE StudentDocumentId = p_document.Id AND StudentDocumentPartitionKey = p_document.DocumentPartitionKey; -- Process addresses array IF p_edfi_doc ? 'addresses' THEN FOR v_address IN SELECT jsonb_array_elements(p_edfi_doc->'addresses') LOOP -- Resolve descriptor references SELECT DescriptorId INTO v_address_type_descriptor_id FROM edfi.Descriptor WHERE namespace_and_codevalue(v_address->'addressTypeDescriptor') = namespace_and_codevalue(Uri); SELECT DescriptorId INTO v_state_descriptor_id FROM edfi.Descriptor WHERE namespace_and_codevalue(v_address->'stateAbbreviationDescriptor') = namespace_and_codevalue(Uri); -- Insert address record INSERT INTO edfi.StudentAddress ( StudentDocumentId, StudentDocumentPartitionKey, AddressTypeDescriptorId, StreetNumberName, City, StateAbbreviationDescriptorId, PostalCode ) VALUES ( p_document.Id, p_document.DocumentPartitionKey, v_address_type_descriptor_id, v_address->>'streetNumberName', v_address->>'city', v_state_descriptor_id, v_address->>'postalCode' ); END LOOP; END IF; -- Process extension flattening functions PERFORM dms.process_extension_flattening('students', p_document, p_edfi_doc); END; $$ LANGUAGE plpgsql;

Performance & Synchronization

Trigger-Based Synchronization

CREATE OR REPLACE FUNCTION dms.document_flattening_trigger() RETURNS TRIGGER AS $$ BEGIN IF current_setting('dms.async_flattening', true) = 'true' THEN -- Queue for async processing INSERT INTO dms.flattening_queue ( DocumentId, DocumentPartitionKey, ResourceName, Operation, QueuedAt ) VALUES (NEW.Id, NEW.DocumentPartitionKey, NEW.ResourceName, TG_OP, NOW()); ELSE -- Synchronous flattening PERFORM dms.flatten_document(NEW.Id, NEW.DocumentPartitionKey); END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER document_after_insert_update AFTER INSERT OR UPDATE ON dms.Document FOR EACH ROW EXECUTE FUNCTION dms.document_flattening_trigger();

Batch Processing Queue

  • Asynchronous Processing: Optional async queue for high-volume scenarios
  • Parallel Processing: Multiple workers can process queue concurrently
  • Error Handling: Failed flattening operations logged and retried

-- Performance test example CREATE OR REPLACE FUNCTION dms_test.test_bulk_flattening_performance() RETURNS VOID AS $$ DECLARE v_start_time TIMESTAMP; v_end_time TIMESTAMP; v_duration INTERVAL; BEGIN -- Insert 1000 test student documents v_start_time := clock_timestamp(); FOR i IN 1..1000 LOOP INSERT INTO dms.Document (ResourceName, EdfiDoc, ...) VALUES ('students', generate_test_student_json(i), ...); END LOOP; v_end_time := clock_timestamp(); v_duration := v_end_time - v_start_time; -- Assert performance within acceptable limits ASSERT v_duration < '30 seconds'::INTERVAL, format('Bulk flattening took %s, expected < 30s', v_duration); END; $$ LANGUAGE plpgsql;
-- Data validation test example CREATE OR REPLACE FUNCTION dms_test.test_flattening_data_integrity() RETURNS VOID AS $$ DECLARE v_json_count INTEGER; v_flat_count INTEGER; v_json_checksum TEXT; v_flat_checksum TEXT; BEGIN -- Insert complex student with addresses and extensions INSERT INTO dms.Document (ResourceName, EdfiDoc, ...) VALUES ('students', '{"studentUniqueId":"12345","addresses":[...]}', ...); -- Compare JSON array count vs flattened table count SELECT jsonb_array_length(EdfiDoc->'addresses') INTO v_json_count FROM dms.Document WHERE ResourceName = 'students'; SELECT COUNT(*) INTO v_flat_count FROM edfi.StudentAddress; ASSERT v_json_count = v_flat_count, format('Address count mismatch: JSON=%s, Flat=%s', v_json_count, v_flat_count); -- Verify referential integrity ASSERT NOT EXISTS ( SELECT 1 FROM edfi.StudentAddress sa LEFT JOIN edfi.Student s ON sa.StudentDocumentId = s.DocumentId WHERE s.DocumentId IS NULL ), 'Orphaned address records found'; END; $$ LANGUAGE plpgsql;

Benefits & Risk Analysis

Key Benefits

Zero Core Modifications

New extensions require no changes to core flattening logic - fully dynamic system

Schema Isolation

Extensions use separate schemas, avoiding naming conflicts and improving organization

MetaEd Integration

Fully generated by MetaEd - no manual coding required for extensions

Performance Optimization

Only enabled extensions processed; registry caching for optimal performance

Risk Mitigation Strategies

Technical Risks

  • Dynamic Function Execution: SQL injection prevention through function name validation
  • Extension Conflicts: Priority system with clear precedence rules
  • Performance Impact: Registry caching and query optimization
  • Table Name Collisions: Hash collision detection with original name prefixes

Success Criteria

  • Core Ed-Fi resources flatten correctly to edfi schema
  • Extensions can be added without core changes
  • Dynamic lookup adds < 5ms overhead
  • 100% test coverage for extension framework
